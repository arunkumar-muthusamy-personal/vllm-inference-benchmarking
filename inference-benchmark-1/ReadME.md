# vLLM Inference Benchmark Plan — gpt-oss-20b

## Hardware Profile

| Component | Spec |
|-----------|------|
| GPU | 1x NVIDIA Ada Lovelace (e.g. RTX 6000 Ada / L40S) |
| VRAM | 48 GB |
| RAM | 64 GB |
| Storage | 500 GB NVMe |
| Tensor Parallelism | 1 (fixed, single GPU) |

---

## 1. Test Dataset

### Why a custom dataset?

Standard open datasets (ShareGPT, LMSYS-Chat-1M, Alpaca) are commonly used in vLLM benchmarks. We use a **synthetic mix** derived from ShareGPT prompt distributions so we control exact token lengths and can reproduce results deterministically.

### Dataset Structure

Each test request is a JSON payload sent to:

```
POST http://<hostname>:8000/v1/completions
Content-Type: application/json
```

### Prompt Categories (test_dataset.jsonl)

We define 5 prompt categories covering realistic workloads:

| Category | Prompt Tokens (approx) | Max New Tokens | Use Case |
|----------|------------------------|----------------|----------|
| short | 64 | 128 | Chat / Q&A |
| medium | 256 | 256 | Summarization |
| long | 512 | 512 | Document analysis |
| xl | 1024 | 512 | Long-context reasoning |
| xxl | 2048 | 512 | Max context stress test |

The dataset file `test_dataset.jsonl` (generated by `generate_dataset.py`) defaults to 200 prompts — 40 per category — shuffled to simulate mixed real-world traffic. Use `--count` to generate larger datasets for sustained high-concurrency runs.

## 2. Concurrency Limits

Based on 48 GB VRAM and gpt-oss-20b model size:

| Scenario | Concurrent Requests | Notes |
|----------|---------------------|-------|
| Minimum (baseline) | 1 | Single-stream latency baseline |
| Low | 4 | Warm-up / sanity check |
| Medium | 16 | Typical interactive load |
| High | 32 | Batch / API gateway load |
| Maximum (stress) | 64 | Near-saturation; monitor OOM |

> Rule of thumb: gpt-oss-20b in 4-bit quantization uses ~12–14 GB VRAM for weights, leaving ~34 GB for KV cache. At 512 token context, you can safely hold ~32–48 concurrent sequences in flight. At 2048 tokens, drop to ~16 max.

## 3. Tunable Parameters

Parameters split into two groups: those that require a **server restart** (set in docker-compose.yml) and those passed at **test time** (set in the benchmark script or request body). You never need to touch docker-compose.yml for test-time params.

### 3.1 Server-Side Parameters → docker-compose.yml (requires restart)

These are passed as `command:` args to the vLLM container. Changing any of these means `docker compose down && docker compose up -d` and waiting ~1–2 min for the model to reload.

| Parameter | Description | Values to Test |
|-----------|-------------|----------------|
| `--quantization` | Weight quantization method | `none`, `awq`, `gptq` |
| `--max-model-len` | Max sequence length (prompt + output tokens combined) | `2048`, `4096`, `8192` |
| `--max-num-seqs` | Max concurrent sequences the scheduler holds | `8`, `16`, `32`, `64` |
| `--max-num-batched-tokens` | Max tokens in a single forward pass | `4096`, `8192`, `16384` |
| `--gpu-memory-utilization` | Fraction of VRAM reserved for KV cache | `0.85`, `0.90`, `0.95` |
| `--block-size` | KV cache block size in tokens | `8`, `16`, `32` |
| `--swap-space` | CPU RAM used as KV cache overflow (GB) | `0`, `4`, `8` |
| `--enable-prefix-caching` | Reuse KV blocks for shared prompt prefixes | flag present / absent |
| `--enforce-eager` | Disable CUDA graph capture (slower but lower memory) | flag present / absent |
| LMCache env vars | KV cache offload layer (see configs 05–07) | enabled / disabled |

### 3.2 Test-Time Parameters → run_benchmark.py CLI args

These do NOT require a server restart. Pass them directly when running the benchmark script.

| Parameter | CLI Arg | Description | Default |
|-----------|---------|-------------|---------|
| Concurrency | `--concurrency` | Number of parallel in-flight requests | `16` |
| Config label | `--config` | Label for output file naming and Grafana annotations | `CONFIG-XX` |
| Num prompts | `--num-prompts` | Number of prompts to send. Cycles dataset if larger than dataset size. Omit to run full dataset once. | full dataset |
| Grafana URL | `--grafana-url` | Grafana base URL for posting run annotations | `http://localhost:3000` |
| Grafana key | `--grafana-key` | Grafana API token. If omitted, falls back to `admin:admin` | see note below |

> `--grafana-key` is optional. By default the script uses `admin:admin` basic auth, which matches the docker-compose setup. Only set this if you've locked down Grafana with a service account token (Grafana → Administration → Service Accounts → Add token). If Grafana is unreachable the annotation is silently skipped.

### 3.3 Per-Request Parameters → test_dataset.jsonl

These are baked into each prompt record in the dataset file. Edit `generate_dataset.py` and regenerate if you want to change them.

| Parameter | Description | Values to Test |
|-----------|-------------|----------------|
| `max_tokens` | Max output tokens per request | `128`, `256`, `512` |
| `temperature` | Sampling temperature | `0.0` (greedy), `0.7` |
| `top_p` | Nucleus sampling cutoff | `1.0`, `0.9` |
| `stream` | Streaming vs batch response | `false` (batch for benchmarking) |

## 4. Configuration Sets

Each config set is a named combination of the above parameters. All sets use `tensor_parallel_size=1`.

For each config, two things are shown:
- **docker-compose.yml `command:` block** — copy this into your compose file, then restart the stack
- **Test command** — run this after the server is up, no restart needed between concurrency levels

---

### CONFIG-01 — Baseline (No Quantization, No LMCache)

**Goal:** Establish raw FP16 throughput and latency baseline.

**→ docker-compose.yml `command:` (restart required)**
```yaml
command: >
  --model /models/gpt-oss-20b
  --port 8000
  --max-model-len 4096
  --max-num-seqs 16
  --max-num-batched-tokens 8192
  --gpu-memory-utilization 0.90
  --block-size 16
  --swap-space 0
  --served-model-name gpt-oss-20b
```

**→ Test commands (no restart needed between these)**
```bash
python run_benchmark.py --host localhost --port 8000 --concurrency 1  --config CONFIG-01 --num-prompts 200
python run_benchmark.py --host localhost --port 8000 --concurrency 16 --config CONFIG-01 --num-prompts 200
python run_benchmark.py --host localhost --port 8000 --concurrency 32 --config CONFIG-01 --num-prompts 200
```

## 5. Metrics to Capture Per Run

| Metric | Description |
|--------|-------------|
| Throughput (req/s) | Requests completed per second |
| Throughput (tok/s) | Output tokens generated per second |
| TTFT (ms) | Time to first token — p50, p90, p99 |
| E2E Latency (ms) | Full request latency — p50, p90, p99 |
| GPU Utilization (%) | From DCGM exporter |
| GPU Memory Used (GB) | Peak VRAM during run |
| CPU RAM Used (GB) | Peak RAM during run |
| Cache Hit Rate (%) | LMCache hit ratio (configs 05–07) |
| OOM / Error Count | Any failed requests |

## 6. Test Execution Plan

### Step 0 — One-time setup (do this once)

```bash
# Install the benchmark script dependency
pip install aiohttp

# Generate the default dataset (200 prompts, 40 per category)
python generate_dataset.py

# Or generate a larger dataset for high-concurrency runs
python generate_dataset.py --count 1000
# Output: test_dataset.jsonl
```

### Step 1 — For each config set

1. Copy the config's `command:` block into `docker-compose.yml` under the `vllm` service
2. If the config has LMCache `environment:` vars, add those too
3. Restart the stack and wait for the model to load:

```bash
docker compose down && docker compose up -d
# Watch logs until you see "Application startup complete"
docker logs -f vllm
```

### Step 2 — Run the benchmark (no restart needed between concurrency levels)

```bash
# Run full dataset at different concurrency levels
python run_benchmark.py --host localhost --port 8000 --concurrency 1  --config CONFIG-02
python run_benchmark.py --host localhost --port 8000 --concurrency 16 --config CONFIG-02
python run_benchmark.py --host localhost --port 8000 --concurrency 32 --config CONFIG-02

# For high-concurrency runs, send more prompts for a longer sustained test
python run_benchmark.py --host localhost --port 8000 --concurrency 32 --config CONFIG-03 --num-prompts 500
python run_benchmark.py --host localhost --port 8000 --concurrency 64 --config CONFIG-03 --num-prompts 1000

# Optional: specify Grafana URL if not on localhost, or use an API token
python run_benchmark.py --host localhost --port 8000 --concurrency 16 --config CONFIG-02 \
  --grafana-url http://localhost:3000 --grafana-key eyJrIjoiT0tTcG1pUlY2...
```

Each run saves a `results_CONFIG-XX_cYY.json` file in the same folder.

### Step 3 — Collect GPU metrics

While the benchmark runs, Prometheus + DCGM are already scraping GPU stats. Check Grafana at `http://localhost:3000` or query Prometheus directly:

```
# GPU utilization
DCGM_FI_DEV_GPU_UTIL

# VRAM used
DCGM_FI_DEV_FB_USED
```

### What run_benchmark.py does (vs vLLM's built-in)

`run_benchmark.py` is the right tool here. It:
- Reads your controlled `test_dataset.jsonl` (known token sizes, reproducible)
- Sends requests with the exact concurrency level you specify
- Reports req/s, tok/s, and latency percentiles
- Saves a JSON result file per run

vLLM's built-in `benchmarks/benchmark_serving.py` (inside the container) uses a request-rate model (req/sec) rather than concurrency, and requires the ShareGPT dataset. It's useful for open-loop load testing but harder to compare apples-to-apples across configs. Stick with `run_benchmark.py` for this benchmark plan.

## 7. Results

Results are captured in two ways — no manual table maintenance needed.

**Per-run JSON files** — `run_benchmark.py` saves a `results_CONFIG-XX_cYY.json` after every run with throughput, latency percentiles, and error count. Use these for the writeup.

**Grafana dashboard** — live and historical metrics for GPU util, VRAM, TTFT, latency, KV cache, and throughput. Each run is annotated automatically so you can zoom into any run window. Open `http://localhost:3000`.

To generate a summary table from all result files after testing:

```bash
python summarize_results.py
```
