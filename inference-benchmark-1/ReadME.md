# vLLM Inference Benchmark Plan — gpt-oss-20b

## Overview

Systematic benchmarking suite for evaluating gpt-oss-20b inference performance across 8 configuration profiles, measuring throughput, latency, and GPU efficiency under realistic workloads. Includes controlled dataset generation, automated Grafana annotation, per-run JSON result tracking, and reproducible analysis.

**Key metrics:** 
- Throughput (req/s, tok/s) 
- Time to First Token (TTFT) — p50/p90/p99
- End-to-End Latency (p50/p90/p99)
- GPU utilization and memory
- KV cache hit rate (where applicable)

**Use this to:** 
- Select optimal vLLM configurations for your deployment scenario
- Quantify throughput/latency tradeoffs of quantization, batching, and caching
- Validate performance improvements before production rollout
- Make data-driven cost optimization decisions

**Quick start:**
```bash
pip install aiohttp
python generate_dataset.py
python run_benchmark.py --host localhost --port 8000 --concurrency 16 --config CONFIG-01
```

## Test Dataset

### Why a Custom Dataset?

Standard open datasets (ShareGPT, LMSYS-Chat-1M, Alpaca) are commonly used in vLLM benchmarks but have uncontrolled token distributions. We use a **synthetic mix** derived from ShareGPT prompt distributions so we can:
- Control exact token lengths
- Reproduce results deterministically
- Isolate performance changes to config tweaks, not dataset variance

### Dataset Structure

Each test request is a JSON payload sent to:

```
POST http://<hostname>:8000/v1/completions
Content-Type: application/json
```

### Prompt Categories (test_dataset.jsonl)

We define 5 prompt categories covering realistic workloads:

| Category | Prompt Tokens (approx) | Max New Tokens | Use Case |
|----------|----------------------|-----------------|----------|
| short | 64 | 128 | Chat / Q&A |
| medium | 256 | 256 | Summarization |
| long | 512 | 512 | Document analysis |
| xl | 1024 | 512 | Long-context reasoning |
| xxl | 2048 | 512 | Max context stress test |

The dataset file `test_dataset.jsonl` (generated by `generate_dataset.py`) defaults to **200 prompts** — 40 per category — shuffled to simulate mixed real-world traffic. 

**Generate the dataset:**
```bash
# Default: 200 prompts (40 per category)
python generate_dataset.py

# For sustained high-concurrency runs, use more prompts
python generate_dataset.py --count 1000
```

## Concurrency Limits

Based on 48 GB VRAM and gpt-oss-20b model size:

| Scenario | Concurrent Requests | Notes |
|----------|-------------------|-------|
| Minimum (baseline) | 1 | Single-stream latency baseline |
| Low | 4 | Warm-up / sanity check |
| Medium | 16 | Typical interactive load |
| High | 32 | Batch / API gateway load |
| Maximum (stress) | 64 | Near-saturation; monitor OOM |

### Memory Calculation

**Rule of thumb:** gpt-oss-20b in 4-bit quantization uses ~12–14 GB VRAM for weights, leaving ~34 GB for KV cache.

```
KV cache per request = 2 × (seq_length × hidden_dim × 2 bytes)

At 512 token context:  Can safely hold ~32–48 concurrent sequences
At 2048 token context: Drop to ~16 max concurrent sequences
```

Monitor with: `nvidia-smi` (watch VRAM) or Prometheus DCGM exporter

## Quick Config Selection

**What is your primary goal?**

```
├─ Max throughput (req/s)?           → CONFIG-06 (AWQ + LMCache + batch=64)
├─ Min latency (p99 < 100ms)?        → CONFIG-07 (AWQ + LMCache + eager mode)
├─ Establish baseline (pure FP16)?   → CONFIG-01 (no quantization)
├─ Long context (8K tokens)?         → CONFIG-08 (max-model-len=8192)
├─ No LMCache available?             → CONFIG-03 (high throughput) or CONFIG-04 (low latency)
├─ Research quantization impact?     → CONFIG-01 vs CONFIG-02 (awq)
├─ Repeated prompts/prefixes?        → CONFIG-05, 06, or 07 (enable LMCache)
└─ Memory constrained?               → CONFIG-04 or CONFIG-07 (smaller batch, eager)
```

## Tunable Parameters

Parameters split into three groups:

1. **Server-side** (docker-compose.yml) — requires restart
2. **Test-time** (run_benchmark.py CLI) — no restart needed
3. **Per-request** (test_dataset.jsonl) — baked into dataset

### 3.1 Server-Side Parameters → docker-compose.yml (requires restart)

These are passed as `command:` args to the vLLM container. Changing any of these means:
```bash
docker compose down && docker compose up -d
# Wait ~1–2 min for the model to load
docker logs -f vllm
```

| Parameter | Description | Why It Matters | Values to Test |
|-----------|-------------|----------------|-----------------|
| `--quantization` | Weight quantization method | Reduces memory & improves throughput; may impact accuracy | `none`, `awq`, `gptq` |
| `--max-model-len` | Max sequence length (prompt + output combined) | Constrains KV cache; affects concurrency ceiling | `2048`, `4096`, `8192` |
| `--max-num-seqs` | Max concurrent sequences the scheduler holds | Too low = underutilization; too high = OOM | `8`, `16`, `32`, `64` |
| `--max-num-batched-tokens` | Max tokens in a single forward pass | Larger = higher throughput but higher latency; memory constrained | `4096`, `8192`, `16384` |
| `--gpu-memory-utilization` | Fraction of VRAM reserved for KV cache | Higher = more cache; risk of OOM if too aggressive | `0.85`, `0.90`, `0.95` |
| `--block-size` | KV cache block size in tokens | Smaller = more fragmentation; larger = potential memory waste | `8`, `16`, `32` |
| `--swap-space` | CPU RAM used as KV cache overflow (GB) | Slower but allows higher concurrency; use only if needed | `0`, `4`, `8` |
| `--enable-prefix-caching` | Reuse KV blocks for shared prompt prefixes | Huge win if you have repeated system prompts | flag present / absent |
| `--enforce-eager` | Disable CUDA graph capture (slower but lower memory) | Lower memory, higher latency; useful for latency-critical workloads | flag present / absent |
| LMCache env vars | KV cache offload layer (CPU/disk) | Extends cache beyond GPU VRAM; useful for long-context scenarios | enabled / disabled |

### 3.2 Test-Time Parameters → run_benchmark.py CLI args

These do **NOT** require a server restart. Pass them when running the benchmark script.

| Parameter | CLI Arg | Description | Default |
|-----------|---------|-------------|---------|
| Concurrency | `--concurrency` | Number of parallel in-flight requests | 16 |
| Config label | `--config` | Label for output file naming and Grafana annotations | CONFIG-XX |
| Num prompts | `--num-prompts` | Number of prompts to send. Cycles dataset if larger. | full dataset (200 or 1000) |
| Grafana URL | `--grafana-url` | Grafana base URL for posting run annotations | http://localhost:3000 |
| Grafana API key | `--grafana-key` | Grafana API token. If omitted, uses basic auth (admin:admin) | admin:admin |

**Grafana auth notes:**
- Default: `admin:admin` basic auth (matches docker-compose setup)
- To use a service account token: Grafana → Administration → Service Accounts → Add token → set `--grafana-key`
- If Grafana is unreachable, annotation is silently skipped (benchmark continues)

### 3.3 Per-Request Parameters → test_dataset.jsonl

These are baked into each prompt record in the dataset file. Edit `generate_dataset.py` and regenerate if you want to change them.

| Parameter | Description | Values to Test |
|-----------|-------------|-----------------|
| `max_tokens` | Max output tokens per request | `128`, `256`, `512` |
| `temperature` | Sampling temperature (higher = more randomness) | `0.0` (greedy), `0.7` (stochastic) |
| `top_p` | Nucleus sampling cutoff | `1.0` (no filtering), `0.9` (nucleus) |
| `stream` | Streaming vs batch response | `false` (batch for deterministic benchmarking) |

### What run_benchmark.py Does (vs vLLM's Built-in)

**Use `run_benchmark.py` for this benchmark.**

| Aspect | run_benchmark.py | vLLM's benchmark_serving.py |
|--------|------------------|---------------------------|
| Dataset | Your controlled `test_dataset.jsonl` | ShareGPT (uncontrolled variance) |
| Concurrency model | **Concurrency** (fixed in-flight requests) | Request rate (req/sec) — harder to control |
| Results output | **JSON per run** (easy to analyze) | Stdout + CSV |
| Reproducibility | Deterministic (fixed token sizes) | Variable (depends on dataset distribution) |
| Best for | Controlled config comparison | Open-loop load testing |

For this benchmark plan, **run_benchmark.py is the right tool**. It lets you:
- Reproduce results across configs without dataset variance confounding results
- Compare apples-to-apples (e.g., CONFIG-01 vs CONFIG-02 at same concurrency)
- Generate JSON results for automated analysis

## Interpreting Results

### Anatomy of a results_CONFIG-XX_cYY.json File

```json
{
  "config": "CONFIG-02",
  "concurrency": 16,
  "num_prompts": 200,
  "duration_seconds": 45.2,
  "throughput_req_s": 20.5,
  "throughput_tok_s": 5240,
  "latency_ms": {
    "p50": 680,
    "p90": 920,
    "p99": 1100
  },
  "ttft_ms": {
    "p50": 35,
    "p90": 42,
    "p99": 48
  },
  "gpu_util_avg": 92,
  "gpu_memory_peak_gb": 39.2,
  "error_count": 0,
  "errors": []
}
```

### Key Metrics Explained

**throughput_req_s (requests per second)**
- How many requests your API can handle
- Higher is better, but not at the cost of latency SLA
- Use for: Capacity planning (e.g., if you need 100 req/s, which config gets there?)

**throughput_tok_s (output tokens per second)**
- True measure of compute efficiency
- Compare across configs: if CONFIG-02 uses 30% less VRAM but maintains same tok/s, that's a win
- Use for: Cost optimization (lower VRAM = smaller GPU instance needed)

**TTFT p99 (Time to First Token, 99th percentile)**
- How long users wait before seeing first response token
- Critical for interactive/chat workloads
- SLA: typically want p99 < 100ms for interactive, < 50ms for real-time
- Use for: Choosing latency-critical configs (CONFIG-04, 07)

**E2E Latency p99 (End-to-End, 99th percentile)**
- Total time from request to response completion
- TTFT + generation time
- Critical for batch workloads
- Use for: Choosing batch configs (CONFIG-03, 06)

**gpu_util_avg**
- Average GPU utilization during run
- If < 50% at high concurrency: you've hit a bottleneck (compute bound)
- If = 100% and latency is acceptable: you're using the GPU efficiently
- Use for: Detecting if your config is balanced or imbalanced

**error_count**
- OOM crashes, request timeouts, or server errors
- Should always be 0. If > 0, reduce concurrency or memory utilization

### Decision Framework

Use this table to interpret your results:

| Observation | Interpretation | Next Step |
|-------------|-----------------|-----------|
| TTFT increases with quantization | Quantization adds latency overhead | Use low-latency quantization (GPTQ or INT8) or accept tradeoff |
| Throughput doesn't improve with LMCache | Your prompts don't share prefixes | Disable LMCache, use budget for larger batch |
| GPU util drops at high concurrency | Compute bottleneck, not memory | Can't improve further; need more GPUs |
| GPU util = 100%, latency acceptable | Good balance | This config is optimal for your workload |
| Error count > 0 at c=64 | OOM at high concurrency | Use c=32 instead, or reduce `--max-num-seqs` |
| p99 latency spikes with batching | Batching causes queueing | Choose low-latency config (CONFIG-04, 07) instead |
| tok/s the same across configs | Throughput bottleneck, not memory | Adjust `--max-num-batched-tokens` or add more concurrency |

## Troubleshooting

### OOM (Out of Memory) on Startup

**Error:** `RuntimeError: CUDA out of memory`

**Diagnosis:**
```bash
# Check available VRAM
nvidia-smi
# Note the "Free" column under VRAM

# Check vLLM logs
docker logs vllm | tail -20
```

**Fix:**
1. Reduce `--max-num-seqs` in docker-compose.yml (e.g., 64 → 32)
2. Reduce `--gpu-memory-utilization` (e.g., 0.95 → 0.85)
3. Reduce `--max-num-batched-tokens` (e.g., 16384 → 8192)

Then restart:
```bash
docker compose down && docker compose up -d
docker logs -f vllm
```

---

### OOM During Benchmark (Not on Startup)

**Error:** Benchmark runs for 30 sec, then crashes with CUDA OOM

**Diagnosis:**
```bash
# Watch GPU memory in real-time
watch -n 1 nvidia-smi
# If memory creeps up to 100% during the run, you've hit a growth leak
```

**Fix:**
1. Reduce concurrency: `--concurrency 32` → `--concurrency 16`
2. Reduce `--num-prompts` to shorten the run
3. If consistently happens, the config is unstable; adjust server-side params

---

### Benchmark Runs Very Slowly

**Symptoms:** Throughput appears to be 1–2 req/s instead of expected 10–20 req/s

**Diagnosis:**
```bash
# Check if GPU is being used
nvidia-smi
# If GPU util is 0%, vLLM is not running on GPU

# Check vLLM logs
docker logs vllm | grep -i "cuda\|gpu\|device"

# Test vLLM directly
curl http://localhost:8000/v1/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "gpt-oss-20b", "prompt": "Hello", "max_tokens": 10}'
# If slow, vLLM is CPU-only
```

**Fix:**
1. Verify vLLM container has GPU access: `docker inspect vllm | grep -i "runtime\|nvidia"`
2. Check `docker-compose.yml` has `runtime: nvidia` or `gpus: all`
3. Verify model weights are downloaded to `/models/gpt-oss-20b`

```bash
# Pre-download model outside container
docker run --rm --gpus all -v /models:/models \
  vllm/vllm-openai:latest \
  --model /models/gpt-oss-20b \
  --load-format auto \
  --download-dir /models
```

### Grafana Annotations Not Showing

**Symptoms:** Benchmark runs, but no annotations appear in Grafana timeline

**Diagnosis:**
```bash
# Check Grafana is running
curl http://localhost:3000

# Check API key works
curl -H "Authorization: Bearer <KEY>" http://localhost:3000/api/auth/identity
```

1. If using default `admin:admin`, ensure Grafana hasn't been locked down with service accounts
2. Generate a service account token:
   - Grafana → Administration → Service Accounts → Add token
   - Pass token: `--grafana-key eyJrIjoiT0tTcG1pUlY2...`
3. If Grafana is unreachable, annotation fails silently; benchmark still completes

### Inconsistent Results Between Runs

**Symptoms:** Same config, same concurrency, but results vary by ±30%

**Diagnosis:**
```bash
# Check for background GPU workloads
nvidia-smi
# Kill any other users' jobs

# Check system load
top -1 | head -15
```

**Fix:**
1. Use fixed `--num-prompts` instead of cycling full dataset
2. Wait 2–3 min between runs for GPU memory to settle
3. Kill other workloads: `pkill -f "python.*benchmark"` 
4. Reboot the system if memory fragmentation is suspected
5. For each config, run at least 2 times and average results

---

### High Error Count in Results

**Symptoms:** `error_count > 0` in results_*.json

**Diagnosis:**
```bash
# Check vLLM logs for errors
docker logs vllm | grep -i "error\|exception" | tail -20

# Check benchmark script logs
python run_benchmark.py ... 2>&1 | grep -i "error"
```

**Fix:**
1. If timeout errors: increase `--timeout` in run_benchmark.py
2. If OOM: reduce concurrency or adjust server params
3. If network errors: ensure vLLM container is reachable from benchmark host

## Key Decisions & Tradeoffs Summary

Use this table to choose your production config:

| Use Case | Recommended Config | Rationale |
|----------|-------------------|-----------|
| **Highest throughput (batch workload)** | CONFIG-06 (AWQ + LMCache + batch=64) | 45–55 req/s with acceptable latency |
| **Lowest latency (interactive chat)** | CONFIG-07 (AWQ + LMCache + eager) | p99 TTFT < 30ms, eager mode |
| **Baseline reference** | CONFIG-01 (FP16 no quant) | Establish upper bound; compare others against this |
| **Long context workloads** | CONFIG-08 (AWQ, 8K context) | Stress test at max sequence length |
| **Memory constrained** | CONFIG-04 (AWQ + eager) | Minimal VRAM usage |
| **Best throughput without LMCache** | CONFIG-03 (AWQ + large batch) | If LMCache unavailable or doesn't help |
| **Best latency without LMCache** | CONFIG-04 (AWQ + eager) | If LMCache unavailable |
| **Research quantization** | CONFIG-01 vs CONFIG-02 | Isolate quantization impact |
| **Cost-optimized (mixed workload)** | CONFIG-02 or CONFIG-05 | Balance of throughput, latency, and resource usage |

### Cost Calculation Example

**Scenario:** You need to serve 100K requests/day with p99 latency < 100ms

```
CONFIG-04 (low latency):  6–8 req/s per GPU
  → Need ⌈100,000 / 86,400 sec × (1 / 7) ⌉ ≈ 2 GPUs
  → Cost: 2 × L40S × $15K/month ≈ $30K/month

CONFIG-06 (high throughput):  45–55 req/s per GPU
  → Need ⌈100,000 / 86,400 sec × (1 / 50) ⌉ ≈ 1 GPU (with headroom)
  → But p99 latency may exceed 100ms; need to verify
  → If latency is acceptable: 1 × L40S × $15K/month ≈ $15K/month

Decision: Run CONFIG-06 at your target concurrency; if p99 < 100ms, use CONFIG-06. Otherwise, use CONFIG-04.
```
